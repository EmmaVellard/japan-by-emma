# Japan by Emma

A bilingual, photography-led personal travel guide for Japan. The site turns my 2024 PDF notes into a fast, mobile-friendly editorial website in English and French.

Production URL: <https://emmavellard.github.io/japan-by-emma/>

## What is included

- English and French routes under `/en/` and `/fr/`
- One practical guide page per language, with 12 linked topics on the same page
- Six day-by-day itineraries per language
- Initial destination pages for Tokyo, Kyoto, and Osaka
- Planned architecture for recommendations, resources, and photo galleries
- Equivalent-page language switching through stable `translationKey` values
- Static HTML with no client framework and no runtime translation
- Responsive images generated by Astro
- Sitemap, canonical URLs, `hreflang`, Open Graph metadata, structured data, and `robots.txt`
- GitHub Pages deployment through GitHub Actions

The source guide dates from 2024. Prices, schedules, transport products, booking rules, and availability are marked for review instead of being presented as permanently current. French editorial files use `translationReview: true` until Emma has reviewed them.

## Technology

- Astro 7 with strict TypeScript
- Tailwind CSS 4
- Astro content collections with Zod schemas
- Markdown editorial content
- Static site generation
- pnpm

The generated HTML is intentionally ready for a future Pagefind-style static search index without changing the content model. Search is not bundled in the first release.

## Local setup

Install Node.js 24 and pnpm, then run:

```bash
pnpm install
pnpm dev
```

Useful commands:

```bash
pnpm check         # Type-check content and components
pnpm build         # Type-check and create the production site
pnpm preview       # Preview the production build
pnpm format        # Format project files
pnpm format:check  # Check formatting without changing files
```

The local URL includes the GitHub Pages base path: `/japan-by-emma/`.

## Content architecture

```text
src/
├── assets/images/              # Personal photos, grouped by place/topic
├── components/                 # Reusable navigation, cards, maps, callouts, and photo UI
├── content/
│   ├── guide/{en,fr}/          # Practical guide articles
│   ├── itineraries/{en,fr}/    # Structured routes and timelines
│   ├── destinations/{en,fr}/   # Destination notes
│   ├── recommendations/{en,fr}/
│   ├── resources/{en,fr}/
│   └── galleries/{en,fr}/
├── data/                       # Map URLs and planned destinations
├── i18n/                       # Interface text, locale routes, and URL helpers
├── layouts/                    # Shared page shells
├── pages/                      # Static locale-aware route templates
└── styles/                     # Tailwind theme and global editorial styles
```

Schemas live in `src/content.config.ts`. Editorial content belongs in Markdown; reusable interface labels belong in `src/i18n/en.ts` and `src/i18n/fr.ts`.

## Add an English guide topic

Create `src/content/guide/en/example.md`:

```yaml
---
title: Example topic
description: A concise summary for cards and search metadata.
locale: en
translationKey: example-topic
slug: example-topic
order: 13
featured: false
lastUpdated: 2026-07-13
needsReview: false
---
```

Write the article below the frontmatter using Markdown headings, lists, and links.

## Add the French guide topic

Create `src/content/guide/fr/exemple.md` with the same `translationKey` and a localized slug:

```yaml
---
title: Sujet d’exemple
description: Un résumé concis pour les cartes et les métadonnées.
locale: fr
translationKey: example-topic
slug: exemple
order: 13
featured: false
lastUpdated: 2026-07-13
needsReview: false
translationReview: true
---
```

The guide renders every topic on one page and uses `translationKey` as the topic anchor. Set `translationReview: false` after the French text has been reviewed.

## Add a destination

Create one Markdown file in each locale under `src/content/destinations/`. Use the same `translationKey`, but localized slugs.

Required destination fields include:

```yaml
region: Kansai
recommendedDuration: 3 days
highlights: []
mapUrl: 'https://...'
```

Only add detailed recommendations supported by my notes. Leave `highlights` empty when the source material is not ready.

## Add an itinerary

Create paired files under `src/content/itineraries/{en,fr}/`. The timeline is structured so the shared `DayTimeline` component can render it:

```yaml
days: 14
recommendedFor: A first trip to Japan
destinations: [Tokyo, Kyoto, Osaka]
timeline:
  - day: '1'
    title: Arrival in Tokyo
  - day: '2–5'
    title: Tokyo
    note: Optional note.
transportNotes: []
alternatives: []
```

## Add a recommendation

Create paired files under `src/content/recommendations/{en,fr}/`:

```yaml
---
name: Place name
description: Why it is useful to know.
locale: en
translationKey: place-name
slug: place-name
type: restaurant
destination: tokyo
status: visited
mapUrl: 'https://...'
priceRange: ''
featured: false
lastUpdated: 2026-07-13
---
```

Use `status: visited` only when Emma personally tried the place. Otherwise choose `researched` or `to-test`.

## Add personal photographs

Place original images under the matching folder in `src/assets/images/`:

```text
homepage/  tokyo/  kyoto/  osaka/  hokkaido/
food/  transport/  itineraries/
```

Recommended source dimensions:

- Hero: 2400 × 1600 px or larger, landscape
- Destination/card: 1800 × 1350 px or larger, 4:3
- Gallery: at least 1800 px on the long edge
- JPEG for photographs at high quality; PNG only when transparency is required

Astro generates responsive widths, WebP output, intrinsic dimensions, and lazy loading. Hero images load at high priority.

### Publish photographs on “Through my lens”

The first public version intentionally contains only:

```text
src/assets/images/lens/01-kyoto.jpg
src/assets/images/lens/02-osaka.jpg
src/assets/images/lens/03-tokyo.jpg
```

Other photographs in the `lens` folder stay local and are ignored by Git. When another photograph is ready to publish, add its filename to both `src/data/lens.ts` and the lens exceptions in `.gitignore`.

Reference a local image from frontmatter:

```yaml
heroImage: ../../../assets/images/tokyo/shibuya-evening.jpg
heroAlt: Shibuya crossing in Tokyo at night
heroCaption: Shibuya, Tokyo
```

Add a localized gallery:

```yaml
gallery:
  - image: ../../../assets/images/kyoto/fushimi-inari.jpg
    alt: Torii gates at Fushimi Inari shrine
    caption: Fushimi Inari early in the morning
    location: Kyoto
    date: 2024-10-18
```

French files may use the same image with different `alt` and `caption` text.

Good alt text describes what matters in the image without starting with “image of.” Captions can add personal context, place, or date; they should not merely repeat the alt text.

## Update the maps

Edit `src/data/maps.ts`:

```ts
export const maps = {
  googleMyMaps: 'https://...',
  mapstr: 'https://...',
} as const;
```

Map embeds are intentionally represented by fast, accessible cards with fallback links. This avoids third-party iframe failures and keeps mobile pages light.

## GitHub Pages deployment

The workflow at `.github/workflows/deploy.yml` builds and publishes every push to `main`.

1. Push the `main` branch to `emmavellard/japan-by-emma`.
2. On GitHub, open the repository and choose **Settings → Pages**.
3. Under **Build and deployment**, set **Source** to **GitHub Actions**.
4. Open the repository’s **Actions** tab and wait for **Deploy to GitHub Pages** to finish successfully.
5. Visit <https://emmavellard.github.io/japan-by-emma/>. The first deployment can take a few minutes to become visible.

Every later push to `main` republishes the site automatically. The workflow can also be started manually from **Actions → Deploy to GitHub Pages → Run workflow**.

The default deployment settings are in `astro.config.mjs`:

```text
SITE_URL=https://emmavellard.github.io
BASE_PATH=/japan-by-emma
```

They can be overridden during a build through environment variables.

## Use a custom domain later

1. Add the domain to `public/CNAME` as a single line.
2. Set `SITE_URL` to the full custom origin, such as `https://japan.example.com`.
3. Set `BASE_PATH` to an empty value.
4. Update the sitemap URL in `public/robots.txt`.
5. Configure the domain's DNS and custom-domain setting in GitHub Pages.

Run `pnpm build` before publishing any structural or content-model change.
